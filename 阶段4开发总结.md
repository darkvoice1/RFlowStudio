# 阶段4开发总结（基础统计分析 MVP）

## 1. 阶段目标与完成情况

### 阶段目标
- 目标1：完成基础统计分析主链路，让用户可以基于同一个数据集发起描述统计、相关分析、卡方检验、独立样本 t 检验和单因素方差分析。
- 目标2：统一统计分析结果结构，让前端和后续导出能力都围绕同一套结果模型工作。
- 目标3：把统计逻辑正式切到 R 执行，并补齐分析历史记录保存与重新运行能力。

### 完成结果（对照目标逐条写）
- 已完成：打通 `POST /api/v1/datasets/{dataset_id}/analysis-jobs` 主链路，支持用户为同一数据集发起五类基础统计分析任务。
- 已完成：分析前会统一读取数据集、应用已启用的数据清洗步骤、校验字段和分组变量，再进入统计执行层。
- 已完成：统计结果统一整理为摘要、表格、图形、说明文本等结构，前端只需要消费统一结果模型。
- 已完成：Python 侧目前负责接收请求、准备数据、下发分析参数，具体统计逻辑统一交给独立的 R 服务执行。
- 已完成：分析完成后会把结果保存到数据库，形成可查询的历史分析记录。
- 已完成：新增历史记录查询和重跑能力，用户可以基于同一个数据集重新运行历史分析。
- 部分完成：路线图中的“结果支持复制、导出和保存”目前已完成“保存”，复制和导出暂未进入本阶段交付。

- 核心实现点：围绕 `DatasetAnalysisService`、`DatasetAnalysisExecutionService`、`DatasetAnalysisRExecutionService` 建立“请求校验 -> 数据准备 -> R 执行 -> 结果入库 -> 历史复跑”的分析主链路。
- 对应功能结果：后端已经可以创建统计分析任务、返回统一分析结果、保存历史记录，并支持用户重新运行历史分析。

- 核心实现点：无。
- 对应功能结果：无。

### 数据/任务/接口实现（按阶段实际选填）
- 数据模型/迁移：新增 `dataset_analysis_records` 表，用于持久化保存分析类型、变量列表、分组变量、分析选项、结果快照和所属任务 ID。

## 3. 新使用技术栈简要介绍

- `Rscript`：本阶段首次正式承担统计分析执行职责，用于承接描述统计、相关分析、卡方检验、t 检验和方差分析等实际计算。
- `plumber`：用于把 R 统计逻辑封装成 HTTP 服务，让 Python 后端可以用统一接口把“清洗后的数据 + 分析参数”转交给 R 执行。
- `Docker Compose` 下的独立 `r-analysis` 容器：用于隔离 R 运行环境，避免本机和 Python 后端环境耦合，同时让后续部署结构更清晰。

## 4. 遇到的问题与解决方案

### 问题1：统计分析最初是 Python 本地实现，后续切 R 时会出现双轨维护风险
- 现象：如果同时保留 Python 统计实现和 R 统计实现，后续每新增一个分析方法都要维护两套逻辑。
- 原因：前期为了先打通主链路，先用 Python 版本完成了结果骨架和基础分析能力。
- 解决：后续统一收口为“Python 负责下发与编排，R 负责实际统计执行”的单轨方案，删除 Python 统计实现分支。
- 防复发措施：后续新增统计分析方法时，默认直接落在 R 侧实现，Python 只保留请求编排、参数校验和结果接入职责。

### 问题2：开发机不希望直接安装完整 R 环境，但分析能力又必须尽快落地
- 现象：本机环境如果直接安装 R，会增加开发环境负担，也不利于后续团队统一运行方式。
- 原因：R 是阶段四新引入的运行时，和现有 Python 后端、PostgreSQL 属于不同技术栈。
- 解决：新增独立 `r-analysis` 容器，把 Rscript 和 `plumber` 服务放进 Compose 编排中，由 Python 通过 HTTP 调用。
- 防复发措施：后续新增 R 依赖时优先更新 `infra/r-analysis/Dockerfile`，保持 R 运行环境只在容器内演进。

### 问题3：统计结果如果只挂在任务结果上，服务重启后历史分析不可复用
- 现象：仅依赖内存任务结果时，用户无法方便地查看历史分析，也无法从历史配置快速重新运行。
- 原因：任务系统当前仍然是轻量级内存实现，不适合承担长期历史结果存储职责。
- 解决：新增 `dataset_analysis_records` 数据表，把已完成分析的请求参数和结果快照持久化到数据库。
- 防复发措施：后续涉及“可回看、可复用、可导出”的分析能力，优先围绕数据库中的历史记录建设，而不是只依赖任务内存态。

## 5. 新增目录与代码文件说明

### 根目录
- `阶段4开发总结.md`：本阶段总结与交接说明。

### 后端目录
- `backend/app/services/dataset/analysis/dataset_analysis_service.py`：负责统计分析请求校验、数据准备、结果落库、历史记录查询和重跑入口。
- `backend/app/services/dataset/analysis/dataset_analysis_execution_service.py`：负责分析执行层编排，把统一准备后的请求转发给 R 执行服务。
- `backend/app/services/dataset/analysis/dataset_analysis_r_execution_service.py`：负责向 R 统计服务发 HTTP 请求并解析响应。
- `backend/app/services/dataset/analysis/r/dataset_analysis_runner.R`：R 侧统计分析主实现，承接五类基础统计分析的实际计算逻辑。
- `backend/app/services/dataset/analysis/r/dataset_analysis_api.R`：R 侧分析接口定义。
- `backend/app/services/dataset/analysis/r/dataset_analysis_api_server.R`：R 侧 API 服务启动入口。
- `backend/app/models/dataset.py`：新增分析历史记录 ORM 模型定义。
- `backend/app/schemas/analysis.py`：定义分析请求、统一结果结构、历史记录响应结构。
- `backend/app/api/v1/datasets.py`：新增分析任务创建、历史记录查询和历史记录重跑接口。
- `backend/app/services/dataset/dataset_service.py`：统一协调数据集、清洗步骤、统计分析任务和历史记录能力。

### 前端目录
- 本阶段未新增前端业务目录。

### 脚本/配置/测试目录
- `infra/r-analysis/Dockerfile`：R 统计服务镜像构建文件。
- `infra/compose.yaml`：把 `r-analysis` 纳入项目统一容器编排。
- `backend/tests/test_analysis_tasks.py`：覆盖分析任务创建、结果保存、历史记录查询、历史记录重跑等主链路测试。
- `backend/tests/test_analysis_execution_backend.py`：覆盖 Python 到 R 服务调用链路及异常处理测试。
